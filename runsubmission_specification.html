Introduction

Link-the-Wiki in 2009: The Link-the-Wiki track aims to produce a standard procedure and metrics for the evaluation of link discovery between documents. Given a new orphan document, the task is to analyse the text and recommend a set of incoming and outgoing links from anchor-text specified as a passage to Best Entry Point (BEP) in the existing collection. This means that anchor text will be linked to a specific position in the target document for starting to read the relevant content. In addition to the INEX 2009 Wikipedia Collection, the Te Ara Encyclopaedia is adopted at the first time in the track. The tasks are to discover links either within the same collection or from the Te Ara documents to the Wikipedia documents. The performance of the link discovery engines will be evaluated.

Link-the-Wiki Task

In 2009, two collections, the INEX 2009 Wikipedia collection and the Te Ara Encyclopedia collection, are adopted and three tasks will be run.
  1. Link-the-Wiki Documents from within the INEX 2009 Wikipedia collection are used. Random selection will be used for file-to-file link discovery. Topic nomination is expected to be used for anchor-to-BEP linking.
  2. Link-Te-Ara The Te Ara Encyclopedia has no links in it. The task is to identify anchor-to-BEP links within the encyclopedia. All topics are used and a number of topics will be chosen for evaluation.
  3. Link-Te-Ara-to-Wiki The task is to link Te Ara to the Wikipedia at anchor-to-BEP level. All topics will be used and a number of topics will be chosen for evaluation.
In link discovery at file-to-file level, participants will be given a set of existing Wikipedia documents, randomly selected up from the collection. The task is to discover incoming and outgoing links. Submissions will be evaluated automatically against the Wikipedia ground-truth using an evaluation methodology, which is similar to the one used in previous years. This task requires no manual assessment.

In link discovery of the Wikipedia at anchor-to-BEP level, topics will be nominated by and distributed to participants. Link engines must identify anchors in the topic document where each anchor can be linked to several best entry points in respective documents. Incoming links are, again, from anchor text in the referring documents to BEP in the topic document. This task will require collaborative assessment by participants in the INEX tradition. An assessment GUI will be used to allow for efficient assessment of links.

Beside the INEX 2009 Wikipedia Collection, the Te Ara Encyclopedia is used in INEX 2009. Currently, there are no links in the collection. Two tasks are designated for the Te Ara Encyclopedia collection. Link-Te-Ara is to achieve anchor-to-BEP linking within the Te Ara collection whilst Link-Te-Ara-to-Wiki dedicates to finding anchors in the Te Ara documents and linking these anchors to respective best entry points in the Wikipedia documents. As a result, all documents in Te Ara will be inter-linked or linked to documents in the Wikipedia. Only outgoing links are necessary for both tasks. A set of submission documents will be significantly selected and distributed to participants for manual assessment.

Assistant Programs

A Java program, XML2FOL, is provided to produce a list of all the element nodes offsets and lengths according to a given XML file. Another Java program, XML2TXT, is also prepared for the participants to convert the XML document into the text-only content. These 2 programs may be used by participants to indicate the precise offset and length for anchors and best entry points. These same programs define the reference in the process of assessment and evaluation. Both anchors and BEPs are specified with respect to the text-only version of the files.

Anchor and Best Entry Point Specification

An anchor is specified in four parts; the filename of the XML document, the start position of the anchor-text, the length of the anchor-text, and the anchor text itself. The first three define the File,Offset,Length (FOL) of the anchor while the latter is used to verify that the anchor text is correctly specified by the FOL. The file name is specified without the path or the extension, the offset specifies the anchor starting byte position within the corresponding text-only document, and the anchor length is specified in bytes.

(file, offset, length)

The file name could be a unique number (e.g. 23816) in the INEX 2009 Wikipedia collection, or a unique name (e.g. Manawatu - entry-6) in the Te Ara collection. The Te Ara encyclopedia does not use a number id to represent a document.

Destination links are specified in two parts: a unique filename and a best entry point (BEP). It is the best location in a document from which a user should start reading the referred material.

(file, offset)

Multiple outgoing links for a sibngle anchor can be specified in the following format in the case of the Wikipedia collection. 
<outgoing>
   <anchor name="Luminiferous aether" offset="1688" length="19">
      <tobep offset="2038">123456</tobep>
      <tobep offset="28">356</tobep>
      ...
   </anchor>
   ...
</outgoing>

Restriction of Linking

The Link-the-Wiki task accepts submissions that identify links from anchor-text to best-entry-points. For instance, a document on Information Retrieval which briefly refers to Latent Semantic Analysis may well link to a document which discusses Dimensionality Reduction. Although the two documents may not seem related, a section on latent semantic analysis in one may link to a section on singular value decomposition in the other.

An anchor, indicated by a combination of Offset and Length, must appear only once in a submission - although it may have multiple distinct BEPs. An anchor-text in one document can be linked to several destinations (BEPs) in other distinct documents. It means that the same set of Offset and Length should not appear more than once and hence there is no duplicated anchor for a given topic. Multiple instances of the same anchor in a document will be ignored in evaluation. Document title can also be an anchor, but like any other anchor it can be linked to at most 5 destinations.
The number of anchors and its BEP links is highly restricted. Up to 50 anchors per topic and up to 5 BEPs per anchor may be specified in anchor-to-BEP link discovery.

By contrast, there is no anchor and BEP indicated in file level link discovery but 250 outgoing and 250 incoming file-level links may be specified.

For evaluation purposes, both anchors, and up to 5 BEPs within each anchor, must be ranked and listed in decreasing order of relevance.

Result Submission

Number of Submissions:
  • Link-the-Wiki:
    • File-to-File (automated assessment, 5,000 topics used)
    • Anchor-to-BEP (manual relevance assessment, 50 topics used)
  • Link-Te-Ara:
    • Anchor-to-BEP (Outgoing only, All topics used)
  • Link-Te-Ara-To-Wiki:
    • Anchor-to-BEP (Outgoing only, All topics used)
File-to-File link discovery for the Wikipedia:
  • Each submission may contain up to 5,000 topics. Missing topics will be regarded as having a score of zero for the purpose of computing system performance.
  • Up to 250 incoming links and up to 250 outgoing links may be specified per topic.
Anchor-to-BEP linking discovery:
  • Each submission may contain up to 50 topics. Missing topics will be regarded as having a score of zero for the purpose of evaluating system performance.
  • Up to 50 anchors per topic and up to 5 BEPs per anchor can be specified. In a single submission, multiple best entry points linked from an anchor text must point to different documents. Up to 250 incoming links can be specified in the case of Link-the-Wiki. Each incoming link must be from a different document.
  • Only outgoing links need to be discovered for the Te Ara tasks in 2009.
Runs that violate requirements and specific rules in any way will be disqualified.

Topics, the Wikipedia collection, and the Te Ara encyclopedia

These two collections with topics are available for download from the INEX 2009 web site (http://www.inex.otago.ac.nz).

Special Rules

Since the topics are not truly orphaned documents, we adopt the following rules to simulate the use-case situation of genuinely new documents:
  1. Participants must delete the topic files from the collection. It is left to the participants to choose the best way to do this. The simplest (and safest) is to physically delete and re-index the collection. Participants my choose virtual deletion instead.
  2. Systems must not use the existing collection links to the orphaned documents. Since it is too complicated to delete the numerous existing links to the topic files from the collection, participants are strictly advised not to use such mark-up. For example, if the Albert Einstein page is a topic file then it is OK to identify the text Albert Einstein in another existing document and suggest a link. It is NOT OK to search for collection links for the Albert Einstein page. In other words, text can be used, but not the knowledge that it appears inside a Wikipedia link element (e.g. ).
  3. Systems are allowed to use any link as long as they are not linking the topic file. In the example above, this means that it is valid to use any tag from the page Special Relativity - with the exception of the links to the topic file itself, namely, links to the Albert Einstein topic file.

Run Timings

Each submission at file level can contain 250 incoming and 250 outgoing links for each of 5,000 topics, which is a total of 2,500,000 links per submission. In the anchor-to-BEP link discovery, the retrieval system needs to identify passages as anchors and BEPs as target links.

Along with the submission of links participants are asked to submit details of the time taken to generate the results (excluding the output to the submission file, but including the reading and parsing of the orphan) and details of the hardware used to generate the result. Time should be CPU time in seconds (to 2 decimal places). The details of the computer are difficult to specify as such details as cache size differs from machine to machine. In 2009 the details of the CPU, core numbers, hyper-threading, and main memory are still requested. Please be as accurate as reasonably possible.

The sample submission format is showed below. 
<details>
   <machine>
      <cpu>Intel Celeron</cpu>
      <speed>1.06GHz</speed>
      <cores>1</cores>
      <hyperthreads>1</hyperthreads>
      <memory>128MB</memory>
   </machine>
   <time>3.04 seconds</time>
</details>
